# 規則 06:防禦性編碼與 MISRA 風格

本節取 MISRA C:2012 / CERT C 的實質原則。專案明確聲明遵循 MISRA 時,引用具體規則編號並按其強制等級 (Mandatory/Required/Advisory) 執行;未聲明時按下列嚴重度執行。

## 6.1 返回值與錯誤處理

- 有返回值的函數,其返回值必須被檢查或顯式忽略(`(void)func();` 並註明原因)。重點:HAL/驅動 API、通訊收發、Flash 操作的返回值被忽略是 MAJOR。(MISRA Rule 17.7)
- 錯誤碼約定全專案一致(0 為成功或枚舉錯誤碼,不得混用),錯誤沿呼叫鏈向上傳遞不得靜默吞掉。
- 錯誤路徑上的資源清理:已獲取的鎖、已使能的外設、已分配的資源,在每條錯誤返回路徑上都要釋放。C 語言中集中清理的 `goto cleanup` 模式是可接受且推薦的。
- 「不可能發生」的分支(switch 的 default、防禦性 else)必須有處置:記錄 + 安全狀態,不得空著。對安全關鍵系統,考慮故障時進入定義好的安全狀態 (fail-safe)。

## 6.2 控制流

- switch 的每個 case 必須 break 或顯式標註 fallthrough(`/* fallthrough */` 或 C23/GCC `[[fallthrough]]`);必須有 default。(MISRA Rule 16.x)
- 單語句的 if/while 也要求大括號。(MISRA Rule 15.6)
- 賦值不得出現在條件表達式中:`if (x = getval())` 按 bug 處理,除非有慣用且加括號的明確意圖。(MISRA Rule 13.4)
- 條件恆真/恆假(型別值域導致、前面已判斷過)提示邏輯錯誤。
- 迴圈必須能證明終止(計數器型別夠寬、退出條件可達;硬體等待迴圈見規則 04 超時要求)。

## 6.3 函數與介面

- 不被外部引用的函數/全域變數必須 `static`(縮小作用域、幫助優化、避免命名衝突)。(MISRA Rule 8.7/8.8)
- 指標參數指向的資料不被修改時必須 `const`;查表/配置資料必須 `const`(放入 Flash 而非佔用 RAM——這在嵌入式同時是正確性和資源問題)。(MISRA Rule 8.13)
- 公開 API 必須驗證參數;模組內部 static 函數可依賴呼叫端契約,但契約要成立。
- 函數原型必須完整(禁 K&R 式、禁隱式聲明);標頭檔中的聲明與定義一致。
- 禁用 `atoi`/`atol`(錯誤不可檢測),用 `strtol` 系列並檢查 `errno`/endptr。

## 6.4 宏與預處理

- 函數式宏的參數和整體都要加括號:`#define SQ(x) ((x) * (x))`;參數在宏體中出現多次時,傳入有副作用的表達式(`SQ(i++)`)會出錯——能用 `static inline` 函數就不用宏。
- 條件編譯分支必須都能編譯(定期用 CI 編所有配置);`#if` 中使用未定義宏會靜默按 0 處理,重要開關用 `#ifdef` + `#error` 防呆。
- 標頭檔必須有 include guard;不得在標頭檔中定義變數/函數(inline 與模板除外)。

## 6.5 斷言與診斷

- `assert` 在 Release(`NDEBUG`)下消失:斷言的表達式不得有副作用;運行期必須保證的條件要用真實檢查而非 assert。
- 嵌入式的 assert 失敗處理要明確定義(斷點/記錄/復位),不能默認呼叫不存在的 `abort()` 支援。
- HardFault/異常處理器不得是空迴圈了事:至少要保存現場(棧幀、LR、fault 狀態暫存器)供事後分析,量產版本通常應觸發受控復位。
- 日誌/`printf` 在即時路徑上的成本(阻塞式 UART printf 可能耗時數毫秒),檢查是否影響時序。

## 6.6 常量與可讀性

- 魔術數字必須命名:暫存器位元用晶片頭文件的定義,協議常量、超時值、緩衝區大小用具名常量,且相關聯的值(緩衝區大小與協議最大長度)要用表達式關聯而非各寫一份。
- 單位入名:`timeout_ms`、`freq_hz`、`len_bytes`——單位混淆(ms/ticks、字節/字)是嵌入式高頻 bug。
- 同一資訊不得有兩個獨立來源(陣列大小用 `sizeof(arr)/sizeof(arr[0])` 或 `ARRAY_SIZE` 宏,不手寫長度)。
