# 規則 09:執行效率與資源佔用

嵌入式效率審核的三個維度:**時間**(最壞執行時間 WCET、即時路徑延遲)、**空間**(RAM/Flash 佔用)、**功耗**。與桌面不同,平均性能不是重點——即時系統關心的是最壞情況;電池設備關心的是喚醒次數。

## 9.0 審核立場(先讀)

- 效率問題的嚴重度:**破壞時序要求/資源預算的才是 MAJOR 以上**(ISR 超時導致丟資料、RAM 放不下、電池壽命不達標);其餘一律 MINOR/INFO。
- **不做無證據的微優化挑剔**:非熱路徑上的 `i++` vs `++i`、手工展開迴圈這類建議不要提。可讀性優先,優化只針對(a)即時路徑、(b)高頻執行點、(c)已知資源緊張的專案。
- 建議優化時,先問「這段代碼的執行頻率和時間預算是什麼」;答不出來的優化建議降為 INFO 並建議先測量(DWT cycle counter、GPIO 翻轉示波、`perf`)。

## 9.1 即時路徑(ISR / 高頻迴圈 / 控制迴路)

- ISR 和控制迴路內逐項檢查昂貴操作:
  - 除法/求餘:Cortex-M0/M0+ 無硬體除法器(軟體除法數十~上百週期);除以 2^n 改位移,固定除數可轉乘法+移位。
  - 浮點:無 FPU 是軟浮點;有 FPU 的核在 ISR 用浮點牽涉 FPU context 保存(lazy stacking 配置)。
  - `printf`/`snprintf`/字串格式化:數百微秒起跳,即時路徑禁止;日誌應記原始值延後格式化 (deferred logging)。
  - 庫函數的隱藏成本:`math.h` 的 sin/sqrt(可查表/多項式近似替代)、`memcpy` 大塊拷貝(考慮 DMA)。
- **WCET 視角**:即時路徑上的迴圈上界、分支最壞路徑是否有界且已知;帶不定長迴圈(如鏈表遍歷)的 ISR 要標記。
- 臨界區/關中斷時長直接構成其他中斷的延遲上界,臨界區內的操作按上述標準從嚴。

## 9.2 記憶體佔用

- **RAM**:
  - 只讀資料(查表、字串、配置模板)必須 `const`(進 Flash 不佔 RAM;漏 `const` 的大表既佔 Flash 又佔 RAM)。
  - 型別過寬:狀態/標誌用 `uint32_t` 陣列、布林用 int 陣列——大量元素時改窄型別或位圖。
  - 重複緩衝區:多個模組各自的大緩衝區能否複用/共用(注意生命週期不重疊才可共用)。
  - 結構體成員按大小降序排列減少 padding(大量實例的結構體才值得提)。
- **Flash**:
  - 代碼體積殺手:`printf` 全功能版(浮點支援)、C++ iostream/異常/RTTI、模板多型別實例化、`-O0` 編譯。
  - 重複邏輯合併;確認開啟 `-ffunction-sections -fdata-sections` + `--gc-sections`(見規則 07 的 KEEP 注意事項)。
- 大物件傳值:結構體按值傳參/返回造成整份拷貝,超過 2 個字寬的結構體用 const 指標/引用傳遞。

## 9.3 I/O 與資料搬運

- 阻塞式輪詢傳輸大塊資料(逐字節等 UART/SPI 標誌)佔滿 CPU:量大或頻繁時應改中斷/DMA 驅動。
- I2C/SPI 交易的批次化:逐暫存器單次讀 N 次 vs 一次連續讀 N 個暫存器(多數感測器支援位址自增)——每次交易的 start/stop/位址開銷在高頻採樣時是主要成本。
- 資料多次拷貝:接收緩衝→解析緩衝→應用緩衝的鏈條,檢查能否原地解析/零拷貝(指標+長度傳遞)。
- 按位元組搬運字對齊資料:對齊時用字寬存取(或直接 memcpy,編譯器會優化);但注意規則 07 的非對齊存取限制。

## 9.4 功耗(電池/低功耗設備)

- **忙等待是功耗 bug**:主循環空轉輪詢應改為事件驅動 + `WFI`/`WFE`(裸機)、RTOS 阻塞(tickless idle 是否啟用)、`epoll`/阻塞讀(Linux)。
- 喚醒頻率:週期定時器的週期是否必要地短;多個定時器能否對齊喚醒點合併喚醒。
- 外設時鐘管理:用完的外設是否關時鐘;ADC/感測器在空閒期是否進低功耗模式。
- Linux:高頻輪詢 sysfs/proc、短週期 timerfd 會阻止 CPU 進深度 idle;用 `powertop` 驗證喚醒源。

## 9.5 編譯器與優化互動

- 檢查是否有妨礙優化的寫法:不必要的 `volatile`(只有 ISR 共享和暫存器才需要,濫用會禁掉所有相關優化)、跨呼叫的全域變數反覆重讀(編譯器無法快取,熱迴圈中先讀入區域變數)。
- 熱路徑上的間接呼叫(函數指標、C++ 虛函數)阻止內聯;每次呼叫查一次的「回呼鏈」在高頻點展開為直接呼叫。
- 依賴 `-O0` 才正確的代碼是 bug(見規則 02 volatile);性能結論必須基於實際優化等級(`-O2`/`-Os`)。

## 9.6 演算法與資料結構(資源受限視角)

- 資料量有界且小(N < 幾十)時,線性掃描勝過雜湊/樹(無分配、無指標開銷、cache 友好)——**不要**建議在小 N 上引入複雜結構,那是反向優化。
- 資料量會增長的路徑(記錄、連接數、隊列)才檢查複雜度:O(n²) 的插入排序在 n 可能到千級時要標記。
- 遞迴改迭代(同時是規則 01 的堆疊要求)。
- 查表 vs 計算的取捨明確化:表佔 Flash/RAM、計算佔週期,依專案哪個資源緊張決定,不默認查表就是優化。
