# 規則 07:可移植性與嵌入式 C++

## 7.1 對齊與佈局

- 結構體直接映射通訊協議/儲存格式時:
  - padding 隨編譯器/平台變化,必須 `#pragma pack` / `__attribute__((packed))` 或(更穩健)逐欄位序列化。
  - packed 結構體的成員取址再解引用,在不支援非對齊存取的核 (Cortex-M0/M0+) 上觸發 fault;即使支援的核上也可能有性能與原子性問題。
- 強制轉型指標提升對齊要求(`uint8_t*` → `uint32_t*`)是 UB 且在 M0 上 fault:改用 `memcpy`。
- **位元域 (bit-field) 不得用於映射硬體暫存器或協議**:位序、跨單元佈局、存取寬度均由實現定義。暫存器用遮罩+位移操作。

## 7.2 位元組序 (Endianness)

- 通訊協議、多位元組資料的網路收發必須顯式轉換(自寫 `get_be16`/`put_le32` 系列或平台 hton 類函數),不得依賴主機序直接 memcpy 後強轉——除非兩端固定同構且有註明。
- 檢查協議文檔定義的字節序與代碼實現一致(CAN、Modbus 大端居多;許多感測器暫存器是大端而 MCU 是小端)。

## 7.3 編譯器依賴

- 實現定義行為不做假設:`char` 的符號性(ARM 上默認無號,x86 有號——`char c = getchar(); if (c < 0)` 兩平台行為不同)、有號右移、枚舉大小。
- 編譯器擴展(`__attribute__`、`#pragma`、內聯彙編)集中封裝在可移植層/宏中,不散落業務代碼。
- 依賴特定優化行為(如假設某函數會被內聯、假設空迴圈延時不被刪除)的代碼要標記——空迴圈延時應使用 `__NOP()` 迴圈或硬體定時器。

## 7.4 嵌入式 C++(專案使用 C++ 時)

- **禁用/慎用清單**(依專案政策,以下為常見嵌入式 C++ 約束,發現使用時確認專案是否允許):
  - 異常 (`throw`/`try`):多數嵌入式禁用(`-fno-exceptions`);禁用時 `new` 失敗行為、STL 容器錯誤路徑都要重新審視。
  - RTTI(`dynamic_cast`/`typeid`):通常禁用(`-fno-rtti`)。
  - 運行期堆分配的 STL 容器(`std::vector`/`std::string`/`std::map`)在禁堆專案中不可用;可用 `std::array`、`etl::`(Embedded Template Library)或靜態容量的自製容器。
  - `iostream`:代碼體積巨大,嵌入式禁用。
- **靜態初始化順序**:跨編譯單元的全域物件建構順序未定義 (static initialization order fiasco);全域物件的建構函數在 `main` 前執行,此時硬體/RTOS 可能未初始化——全域物件建構函數中不得碰硬體或 RTOS API。建議用顯式 `init()` 或函數內 static(注意其執行緒安全鎖的實現代價)。
- 虛函數:vtable 開銷通常可接受,但要確認純虛呼叫、建構函數中呼叫虛函數的問題;ISR 呼叫路徑上的虛函數會妨礙最壞執行時間分析。
- RAII 是嵌入式 C++ 的核心價值:鎖守衛、關中斷守衛、外設使能守衛,優先於手動配對呼叫。
- `constexpr`/`static_assert` 積極使用:查表、配置驗證移到編譯期,零運行期成本。
- 模板膨脹:對每個型別實例化一份代碼,Flash 受限專案檢查模板使用是否造成體積失控(map 檔比對)。
- 中斷向量表註冊的 C++ 成員函數:ISR 必須是自由函數或 static 成員函數(`extern "C"` 連結),不能是非靜態成員函數。

## 7.5 啟動與連結

- 自訂連結腳本的段放置:代碼假設某變數在特定段 (`.noinit`、CCM RAM、外部 RAM) 時,確認屬性標註與連結腳本一致;放在 CCM/TCM 的緩衝區不能做 DMA(匯流排不可達)是常見陷阱,依平台確認。
- `-ffunction-sections -fdata-sections` + `--gc-sections` 下,只被中斷向量表/彙編引用的符號需要 `KEEP` 或 `used` 屬性,否則被連結器刪除。
