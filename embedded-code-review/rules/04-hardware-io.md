# 規則 04:硬體 I/O 與外設驅動

硬體不遵守軟體的假設:暫存器讀寫有副作用、外設有生效延遲、匯流排會失敗。驅動代碼審核的核心問題是「硬體不配合時,這段代碼會怎樣?」

## 4.1 暫存器存取

- 指向硬體暫存器的指標必須 `volatile`(缺失時編譯器會合併/刪除/重排存取)。
- **讀-改-寫暫存器**:`REG |= BIT` 對某些暫存器是錯的——
  - 狀態暫存器常是「寫 1 清零 (W1C)」:`REG |= BIT` 會把讀到的其他已置位標誌一併清掉,必須直接寫 `REG = BIT`。
  - 有副作用的暫存器(讀即清、FIFO 資料暫存器)不可用 RMW。
  - 對照晶片手冊確認每個暫存器的存取語義;無法確認時在報告中提出疑問。
- 寫入後需要生效時間的操作(時鐘使能後、電源域切換後):檢查是否有必要的等待/回讀確認。常見錯誤:使能外設時鐘後立即存取其暫存器(某些平台需要 dummy read 或幾個週期延遲)。
- 16/32 位暫存器要求對應寬度的單次存取時,確認編譯器不會拆分(用正確寬度的 volatile 型別,不做位元域映射——見規則 07)。

## 4.2 輪詢與超時(高頻問題)

- **每一個等待硬體狀態的迴圈都必須有超時**:`while (!(REG & READY_BIT));` 在硬體故障、訊號線鬆脫、外部晶片未上電時永久掛死。這是 MAJOR 起評,關鍵路徑為 BLOCKER。
- 超時後的處理路徑要合理:返回錯誤碼、記錄、必要時復位外設,而不是照常繼續。
- 超時實現本身檢查回繞安全(見規則 03 的定時器回繞)。

## 4.3 錯誤處理與恢復

- 匯流排操作(I2C/SPI/UART)必須處理失敗:I2C NACK、匯流排卡死 (bus stuck / SDA held low)、仲裁丟失。I2C 卡死的標準恢復是 toggle SCL 9 次——沒有恢復機制的 I2C 驅動要標記。
- 通訊外設錯誤標誌(UART overrun/framing error、SPI overrun、CAN bus-off)必須被讀取和清除,否則某些外設會停止接收。CAN bus-off 需要有恢復策略。
- 外部晶片初始化:檢查是否驗證了 chip ID / WHO_AM_I,失敗時的行為是否明確。

## 4.4 DMA

- 緩衝區生命週期:DMA 進行中緩衝區不得被釋放/覆寫/移出作用域(區域變數做 DMA 緩衝區是 BLOCKER)。
- Cache 一致性(有 D-Cache 的核):TX 前 clean、RX 完成後 invalidate;緩衝區按 cache line(通常 32 位元組)對齊且大小為其倍數,否則 invalidate 會殃及相鄰資料。
- DMA 完成的判定:輪詢標誌或中斷,兩者都要處理半傳輸/錯誤中斷的情況。
- 迴圈模式 (circular) DMA 讀取位置計算(基於 NDTR/剩餘計數)是否處理了回繞。

## 4.5 GPIO 與外部訊號

- 按鍵/機械訊號輸入必須去彈跳 (debounce)。
- GPIO 中斷觸發沿配置與實際電路(上拉/下拉、有效電平)是否一致。
- 輸出腳初始化順序:先設輸出值、再切換為輸出模式,避免瞬間毛刺(對使能腳、繼電器控制尤其重要)。

## 4.6 看門狗與低功耗

- 看門狗餵狗位置:必須在主循環/監控任務中,**不得在定時器 ISR 中無條件餵狗**(主邏輯死掉後 ISR 仍在跑,看門狗形同虛設)。
- 長時間阻塞操作(Flash 擦寫、大量資料傳輸)期間的餵狗策略。
- 進入低功耗模式前:檢查喚醒源已配置、未完成的傳輸已結束、進入前有無競態(喚醒事件在「檢查完畢」與「WFI」之間到達——Cortex-M 上 WFI 對 pending 中斷的行為要確認)。

## 4.7 Flash / NVM 操作

- Flash 寫入前確認已擦除、地址對齊、不越過頁/扇區邊界。
- 執行中的代碼所在 bank 被擦寫的問題(需要 RAM 執行或雙 bank)。
- 掉電安全:配置資料更新是否有斷電損毀防護(雙副本 + CRC、先寫後切換)。寫到一半掉電後,下次啟動讀到什麼?
- Flash 擦寫期間中斷向量/ISR 是否還在 Flash 上可取指。
